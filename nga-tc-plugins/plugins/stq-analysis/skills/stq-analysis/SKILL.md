# SQT 5-Layer 업무 분류 분석

업무 리스트를 입력받아 SQT 5-Layer 방법론으로 자동 분류·판정·스코어링한다.
리더가 제출한 업무 리스트를 NGA 관점에서 분석하여 일관된 결과를 산출한다.

---

## 입력 형식

업무 리스트는 다음 필드를 포함해야 한다:

| 필드 | 필수 | 설명 |
|---|---|---|
| No. | O | 업무 번호 |
| 업무명 | O | 업무 이름 |
| 담당자 | O | 담당자명 |
| 주기 | O | 반복 주기 (매일/매주/격주/매월/분기/비정기) |
| 1회 소요시간 | O | 1회 수행 시 걸리는 시간 |
| AI 전환 가능성 | - | 리더 자가 판정 (있으면 NGA 판정과 비교) |
| 현재 AI 활용 여부 | - | 예/아니오/일부분 |
| 비고 | O | **프로세스 기술. 이 필드가 분석의 핵심** |

---

## 분석 프로세스

아래 5개 Layer를 순서대로 실행한다. **절대 Layer를 건너뛰지 않는다.**

### Layer 1. 업무 기술 품질 체크

비고(프로세스) 필드를 검증한다.

**판정 기준:**

| 등급 | 조건 | 처리 |
|---|---|---|
| ✅ 충분 | "어디서 → 무엇을 → 어떻게 → 어디에" 중 3개 이상 기술됨 | Layer 2로 진행 |
| ⚠️ 부족 | 일부만 기술됨. 추론은 가능하나 불확실 | Layer 2로 진행하되, "프로세스 보완 필요" 코멘트 추가 |
| ❌ 불가 | 프로세스 미기재 또는 추상적 단어만 있음 | **⚠ 판단 불가** 판정. Layer 2~5 스킵 |

**자동 불가 트리거:**
비고 필드에 "분석", "정리", "관리", "진행", "처리", "작성", "확인", "검토" 등의 단어만 있고, 구체적 데이터소스/프로세스/산출물 기술이 없으면 → ❌ 판단 불가

---

### Layer 2. SQT 태깅

각 업무에 S / Q / T 태그를 부여한다. 복수 태그 가능.

**태깅 규칙:**

```
T (Time) 부여 조건 — 아래 중 3개 이상 해당:
  □ 데이터 소스가 고정되어 있다
  □ 처리 로직이 매번 동일하거나 매우 유사하다
  □ 결과물 포맷이 정해져 있다
  □ 담당자가 바뀌어도 프로세스가 동일하다
  □ "인턴한테 매뉴얼 주면 그대로 할 수 있는" 업무다

Q (Quality) 부여 조건 — 아래 중 2개 이상 해당:
  □ 맥락에 따라 결과가 달라진다
  □ 정답이 하나가 아니다 (해석의 여지)
  □ 경험과 직관이 품질을 좌우한다
  □ 이해관계자와의 커뮤니케이션이 포함된다
  □ "같은 데이터를 줘도 사람마다 결과가 다른" 업무다

S (Scale) 부여 조건 — 아래 중 2개 이상 해당:
  □ 처리 대상(셀러 수, 상품 수, 거래량 등)이 증가 추세다
  □ 물량이 늘면 소요시간이 비례 증가한다
  □ 현재는 수기로 가능하지만 2배가 되면 불가능하다
  □ 사람을 더 뽑는 것 외에 해결책이 없는 구조다
```

**혼합 태그 처리:**
하나의 업무 안에 T 구간과 Q 구간이 혼재할 경우 반드시 분리하여 각 구간의 비중을 추정 기술한다.
- 예: "기획전 회고 (6h)" → 데이터 수집·정리 = T(~2h) / 인사이트 도출 = Q(~4h)

---

### Layer 3. AI 전환 가능성 판정

SQT 태그 결과를 기반으로 판정한다.

**판정 규칙:**

```
IF T 태그 포함 AND 입출력 패턴 명확 AND 데이터 소스 고정:
    → ① 즉시 AI 전환 가능

IF T 태그 포함 BUT 기술적 제약 존재 (API 없음, 크롤링 법적 리스크 등):
    → ①△ 조건부 가능

IF Q 태그 포함 OR 프로세스 일부만 반복:
    → ② AI 보조로 효율화

IF 전체가 판단·관계·감정 기반 AND 규칙화 불가:
    → ③ 인간 판단 필수

IF 이미 AI로 완전 운영 중 (소요시간 0 또는 추가 자동화 여지 없음):
    → ✅ 자동화 완료

IF 프로세스 미기재 또는 불충분 (Layer 1에서 ❌):
    → ⚠ 판단 불가
```

**판정 시 주의사항:**

1. **리더가 "가능"이라고 해도 무조건 동의하지 않는다.** 프로세스를 보고 독립 판정한다.
2. **업무 전체를 한 덩어리로 보지 않는다.** 구간별로 분해하여 T/Q를 분리한다.
3. **"중요도 판단", "전략 수립", "방향 설정", "이슈 파악"** 등이 업무의 본질이면 Q축이다.
4. **현재 도구 기준이 아니라 업무 구조 기준으로 판정한다.**

---

### Layer 4. FORCE 검증 — ② 판정 업무 대상

② 판정 업무에 대해 FORCE 질문을 적용하여 안개지식 구간을 특정한다.

| FORCE | 질문 | 검증 목적 |
|---|---|---|
| **F** — Focus | 이 업무는 궁극적으로 어떤 의사결정을 지원하기 위해 존재하는가? | 목적 불분명 → 폐지/통합 검토 |
| **O** — Open | 구성요소를 분해하면 몇 단계로 나뉘는가? | T 구간 분리 가능 여부 확인 |
| **R** — Relate | 다른 업무와 어떻게 연결되어 있는가? | 크로스 부서 중복 식별 |
| **C** — Constrain | 지금 중요한 것과 버릴 것의 경계는? | 자동화 범위 확정 |
| **E** — Evolve | 6개월 후 규모·복잡도는 어떻게 달라지는가? | S 리스크 사전 식별 |

**검증 후 분기:**

```
FORCE O로 분해 →
    IF T 구간 발견됨 → 해당 구간만 ①로 승격
    IF T 구간 없음 → ② 유지

FORCE F 검증 →
    IF 업무 목적 불분명 → "폐지/통합 검토 대상" 코멘트

FORCE E 검증 →
    IF 6개월 내 규모 증가 예상 → S 태그 추가
```

### Layer 5. 임팩트 스코어링 — ① 판정 업무 대상

#### 5-A. 데이터 접근성 필터

**자동 보류(Hold) 조건 — 하나라도 해당 시:**

| 조건 | 판정 |
|---|---|
| API가 없거나 권한 미확보 | ⏸ 보류 |
| 데이터가 폐쇄망/로컬/종이에 존재 | ⏸ 보류 |
| 외부 도구로 데이터 전송이 법적/보안상 불가 | ⏸ 보류 |
| 크롤링이 이용약관 위반 가능성 있음 | ⏸ 보류 |

**접근 가능 기준:**

| 접근 가능 → 스코어링 진행 | 접근 불가 → 보류 |
|---|---|
| 구글시트/엑셀 파일 | 사내 폐쇄망 전용 |
| CSV/JSON 다운로드 가능 | 내보내기 기능 없음 |
| API 열림 + 권한 확보 | API 있지만 권한 없음 |
| 웹에서 복붙 가능 | 개인정보/보안 데이터 |
| Redash/BI 쿼리 가능 | 법적 외부 전송 불가 |

보류 업무는 스코어링하지 않는다. 보류 사유와 재진입 조건을 기술한다.

#### 5-B. 임팩트 스코어링

**자동화 임팩트 점수 = 반복 빈도 × 1회 소요시간 × 담당자 수**

| 요소 | 1점 | 2점 | 3점 | 4점 | 5점 |
|---|---|---|---|---|---|
| 반복 빈도 | 분기 | 매월 | 격주 | 매주 | 매일 |
| 1회 소요시간 | 30분 미만 | 30분~1시간 | 1~2시간 | 2~3시간 | 3시간+ |
| 담당자 수 | 1명 | 2명 | 3명+ | — | — |

**소요시간 미기재 시:** 스코어링 불가. "소요시간 보완 필요" 코멘트.

#### 5-C. 난이도 보정

**보정 점수 = 임팩트 점수 × 난이도 계수**

| 난이도 | 조건 | 계수 |
|---|---|---|
| 단순 | 단일 데이터 소스, 시트 연동, 단일 API | ×1.0 |
| 보통 | 멀티 소스, 조건 분기, 2개 이상 시스템 | ×0.7 |
| 복잡 | 외부 시스템 연동, 권한 이슈, 커스텀 개발 | ×0.4 |

#### 5-D. 타이브레이커 (동점 시)

1. S 태그 있는 업무 우선
2. 크로스 부서 중복 있는 업무 우선
3. 데이터 소스 단순도 높은 업무 우선

## 판정 일관성 규칙

**분석 시 반드시 지키는 10개 규칙:**

1. 리더 판정을 그대로 수용하지 않는다. 프로세스를 보고 독립 판정한다.
2. "가능"이라고 적혀있어도 Q 구간이 본질이면 ②로 내린다.
3. 프로세스가 없으면 무조건 ⚠ 판단 불가. 추측으로 판정하지 않는다.
4. 업무를 구간별로 분해한다. 6시간짜리 업무가 전부 T일 리는 없다.
5. 기술적 제약이 보이면 ①이 아니라 ①△. API 없음, 크롤링 리스크, 권한 미확보.
6. 이미 AI로 완전 운영 중(소요시간 0)이면 ✅ 자동화 완료. ①에 넣지 않는다.
7. 소요시간 미기재 업무는 스코어링하지 않는다.
8. "중요도 판단", "전략 수립", "방향 설정", "이슈 파악"이 본질이면 최소 ②, 경우에 따라 ③.
9. 유사 구조 업무는 클러스터로 묶어 통합 자동화 가능성을 코멘트한다.
10. 보류 ≠ 불가. 보류 업무에는 반드시 재진입 조건을 기술한다.
